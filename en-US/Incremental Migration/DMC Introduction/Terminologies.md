### Logical Replication

This term is in contrast to physical replication. Physical replication refers to directly copying the physical logs from the primary database to the standby database, where the standby database applies the physical logs to synchronize the primary database data. Logical replication refers to sending logical logs (which may be directly generated by the primary database or parsed from physical logs) to the standby database. Logical logs can generally be converted to SQL statements, and the standby database synchronizes the primary database data by applying these SQL statements. Physical logs can be considered as binary stream replication, which can only be applied to homogeneous databases; while logical replication is semantic replication, applicable to heterogeneous databases.

### Online Migration

This term refers to the migration that pursues non-stop or minimal downtime of the source database, which is generally completed through full snapshot + incremental data capture (CDC) technology.

### CDC(Change Data Capture)

A database technology used to monitor and capture data changes in the database, such as inserts, updates, and deletes (usually parsed from its binary logs). These changes are fully recorded and written in the order they occur to other databases or message middleware, allowing other services to subscribe, consume, and analyze these data changes.

### Cutover Time

The downtime required to migrate the business from the source database to the target database.

### Group Commit Technology

Typically, RDBMSs use the Write Ahead Log (WAL) mechanism, where the bottleneck for data writing speed usually lies in writing redo logs. Transaction commits effectively wait for the redo logs to be flushed to disk. Therefore, some RDBMSs use redo log group commit technology to merge multiple redo flush operations into one. At this time, the transactions being flushed must ensure that there are no dependencies between them, and DMC utilizes this feature to support multi-threaded logical replication under transaction consistency mode.

### Transaction Consistency Synchronization

A synchronization method that ensures the execution order consistency of transactions between the source and target databases. This type of synchronization allows unrelated transactions to be processed concurrently. Such a synchronization method is typically suitable for scenarios where immediate consistency is required for transaction queries on the target side.

### Data Consistency Synchronization

To ensure eventual consistency of data after a delay. This type of synchronization theoretically has a higher throughput limit because it does not focus on the execution order of transactions on the source side. Transactions can be split, merged, and reorganized for more efficient concurrent processing. This type of synchronization is typically suitable for data integration computing, data analysis, and other scenarios.

### YStream

A high-performance physical log parsing library provided by YashanDB that can parse database physical logs into logical logs for logical replication.

### Connector

Refers to the entity that consists of heterogeneous data sources (RDBMS, message queues, file systems, third-party applications, etc., capable of providing storage and querying data) and the client applications implemented by DMC that connect to these data sources.

### Resume Transmission

Refers to the capability of DMC to quickly recover and resume tasks after a synchronization task fails due to user manually terminating the task or factors such as power failure or network issues affecting the source, target connectors, or DMC server itself during the incremental data synchronization phase.

### Metadata Synchronization Phase

Refers to the phase of the task performed by DMC after task initialization, also abbreviated as the metadata phase. During this phase, DMC typically completes operations such as querying, synchronizing, and verifying metadata to ensure subsequent phases can proceed smoothly.

### Full Data Synchronization Phase

Refers to the task phase performed by DMC after metadata synchronization, also abbreviated as the full phase. In this phase, DMC typically completes data querying and synchronization, synchronizing snapshot data from the source to the target through logical replication with high performance.

### Incremental Data Synchronization Phase

Refers to the task phase performed by DMC after full data synchronization, also abbreviated as the incremental phase. In this phase, DMC executes a continuous stream job task until the user manually terminates it. This phase connects the accumulated and newly generated data since the source-side metadata synchronization and full data synchronization, synchronizing this data to the target side concurrently through logical replication.

### Synchronization Mode

The synchronization modes of migration tasks include initial mode, metadata mode, snapshot mode, and increment mode. Among them, the initial mode migration task includes the metadata synchronization phase, full data synchronization phase, and incremental data synchronization phase; the metadata mode migration task includes only the metadata synchronization phase; the snapshot mode migration task includes the metadata synchronization phase and full data synchronization phase; the increment mode migration task includes the metadata synchronization phase and incremental data synchronization phase.

### LCR

Refers to the structured data parsed by the source connector.

### Full Fault Tolerance

Only applicable in snapshot mode with YashanDB as the target. In the metadata synchronization phase, full fault tolerance refers to the fact that anomalies in synchronization and verification between tables do not affect each other. In the full synchronization phase, full fault tolerance is divided into table-level fault tolerance and table content errors. Table-level fault tolerance means that synchronization errors within individual tables do not affect the normal synchronization of other tables; table content errors mean that during synchronization, insertion errors will only stop the synchronization of that table if the number of erroneous rows exceeds the fault tolerance threshold.

### Incremental DDL Fault Tolerance

Refers to the automatic ignoring of DDL captured during the incremental data synchronization phase if it contains unsupported data types or default values. It also ignores all subsequent DDL/DML changes for the involved table (until DROP TABLE is captured) to ensure that the migration task can continue to run normally.

### Flashback Point

Refers to the system change number of the source or target database recorded in DMC. The flashback point can ensure transaction consistency between the source and target, and is commonly used for incremental validation during incremental data synchronization.

### Tool Type

To accommodate and transform all supported data types in the connectors, DMC has a built-in set of data types called tool types, along with mappings from different source-side data types to tool types, and from tool types to different target-side data types.

### Data Type Custom Mapping

Refers to the user-defined mapping of tool types to target-side data types in the configuration file using JEXL expressions. The impact scope of data type custom mapping includes: table creation in the metadata phase; table creation, new column types, and modified column types in the incremental DDL phase.

### Blacklist

Refers to the configuration of the scope of tables to be ignored on the source side in the task configuration file. Tables within this scope are added to the blacklist, and any DDL or DML generated by these tables during the task's operation will not be synchronized to the target side.
